## 设计问题定义

工作中一个待办念头闪过时，现有工具（切到 app / 打开窗口 / 选清单或日期）的摩擦足以让人放弃记录或打断心流。需要为单人、在桌面高频工作的人，提供一个常驻桌面边缘、记一条只需"一次击键 + 一行字"、记完即离开视野的极简待办，核心约束是简单、轻量、不打扰、动效好、美观，绝不臃肿。

## 研究基础

来源：`docs/research/2026-06-25-todo-capsule-research.md`（26 Agent 并行竞品深研 + 桌面胶囊技术可行性 + 8 Agent 对抗验证）。

- **最强共识发现**（高置信，跨 4+ 竞品一致）：顶级"工作中快速捕获"产品的主写入路径是**全局热键召唤捕获面到用户面前**（Things 3 / Todoist / Notion Calendar / Superlist），`invoke → type → Enter → gone`；捕获与组织解耦，写入零决策（单一 Inbox 兜底）。
- **关键争议（裁决）**：用户初始诉求是"hover 胶囊写待办"，但研究证据指出打字流中"移鼠标到边缘"本身是摩擦 → 裁决为**双入口、热键为主写 / hover 为辅看**（用户 Phase 3 Q2 确认）。
- **AI Native 可行性判定**：**not_suitable**。这是纯 GUI 操作型产品，无机器判断步；研究 MVP 中唯一带"机器猜测"性质的 NLP 日期/标签解析已被砍除（见下）。
- **技术可行性（对抗验证后）**：桌面边缘胶囊在 macOS **有条件可行**；"无条件悬浮于他人全屏之上"无稳定保证（best-effort）；hover 触发为 best-effort（光标轮询，非 DOM :hover）；Stage 2 推荐原生 SwiftUI/AppKit NSPanel。
- **研究空白**：未做真实用户测试（竞品/技术研究为主）；"扁平无组织"在长清单下的体验拐点缺实测数据（记为 deferred）。

## 用户行为与心智分析

- **用户当前行为**：工作中念头闪过 → 要么切到待办 app（高摩擦、打断心流）、要么记在脑子/便签/备忘录（易丢失）。痛点在"记录这一步的摩擦"，不在"管理任务"。
- **心智类比**：用户主动把静止胶囊类比为 **Granola 的常驻药丸**（用户附图确认）——深色竖向药丸、顶部图标、下方状态位。展开/形变心智类比 **iOS 灵动岛**（一个元素在状态间形变）。
- **认知负担最高点**：捕获那一刻——任何必填项/选择层都会增加负担。故捕获态唯一必填=标题，无清单/日期/项目选择（Phase 3 Q3 确认"纯扁平无日期"）。

## 方案探索

### Approach A：双面 — 保守路径

- **描述**：hover 药丸 → 侧边滑出清单面板（看/勾/拖/补）；全局热键 → 独立小捕获框浮现在光标附近（Things 3 Quick Entry 式），打字→回车→飞入清单→消失。看与写是两个分离表面。
- **核心假设**：用户能接受"看"和"写"是两种不同形态/位置。
- **行为改变**：低——贴近现有 Quick Entry 心智，几乎不需学习。
- **Evaluability**：N/A（非 AI）。
- **Phase 3约束关联**：满足 Q2（双入口）、Q3（扁平）、Q4（药丸静止态）。
- **优点**：最稳、最贴研究先例、实现风险最低。
- **缺点**：两套表面割裂，缺"一个元素"的连续感，动效叙事弱。
- **风险**：平庸，没吃到用户选的"灵动岛药丸"形态的潜力。
- **最适合的条件**：追求最快落地、最低风险时。

### Approach B：统一形变（灵动岛式）— 理想路径 ✅ 选定

- **描述**：全程**只有一个连续形变的元素**。状态机：静止(显示未完成数) → hover 长成清单(可勾/拖/补一条) → 全局热键就地形变成"带聚焦输入框的捕获面板" → 回车后微确认 → 缩回静止。**从不弹独立窗口**。借 iOS 灵动岛编排 + Superlist「surfaces morph rather than swap」。
- **核心假设**：一个会形变的单元素能同时承载"看"与"写"且用户不迷失当前态（Oracle F6 已就此加显式态信号）。
- **行为改变**：中——需建立"热键召唤"肌肉记忆（首次引导降低成本）。
- **Evaluability**：N/A（非 AI）。
- **Phase 3约束关联**：直接落地 Q4（用户选的"灵动岛式药丸"）、Q2（同一元素承载双入口）、Q3（扁平清单）。
- **优点**：最佳"动效+美观+不打扰"，单元素承载全状态，视觉连续。
- **缺点**：形变编排要求实现精细度更高；态过载风险（已由 F2/F6 的焦点机+态信号化解）。
- **风险**：hover 触发脆弱（F3）、焦点交接复杂（F2）——已加去抖+兜底+三态焦点机。
- **最适合的条件**：用户已选定药丸/灵动岛形态、追求"轻量但高级"的体感（本例正是）。

### Approach C：临时召唤 — 非显而易见路径（反转）

- **描述**：反转"常驻面板"——hover/点击不展开侧面板，而是屏幕中央短暂浮现完整清单（命令面板/HUD 式），Esc 收。清单"召之即来、用完即走"。
- **核心假设**：用户更接受"看列表=一次短暂召唤"而非"就地揭示"。
- **行为改变**：中高——与用户"hover 上去看"的直觉相左。
- **Evaluability**：N/A。
- **Phase 3约束关联**：违反 Q2 中"hover 看"的就地揭示预期（已在方案中声明此冲突 → 故被否）。
- **优点**：最克制，中央大视图看长清单更舒展。
- **缺点/风险**：与用户直觉冲突，赌性大。
- **最适合的条件**：若用户更看重"出视野即清净"甚于"就地一瞥"。

**方案约束检查（Phase 4）：**
- [x] 范式转变：A/B/C 均移除了"开 app/找窗口/选清单+日期"的执行步（GUI 层），并以"静止未完成数"作为不展开即可判断负担的瞄一眼判据。（注：本产品无 AI，"判断辅助"为 GUI 层瞥视判据而非 AI 判断。）
- [x] 非显而易见角度：Approach C 来自"常驻面板 → 临时召唤"的反转。

## 方案关系

A / B / C **互斥**（同一交互重心只能选一种编排）。用户选定 **B**；A 作为"若 B 实现精细度风险过高"的降级退路保留；C 已否定。

## 推荐方案

**选择**：Approach B（统一形变·灵动岛式胶囊）
**理由**：直接落地用户主动选定的药丸/灵动岛形态（Q4 + 附图），在"动效好/美观/轻量/不打扰"四项最优，单元素承载全状态符合"绝不臃肿"。其两大风险（hover 触发脆弱 F3、焦点交接 F2）已由"驻留去抖+非hover兜底入口"和"三态焦点机"化解。
**翻转条件**：若 Stage 2 技术 spike 证明"单元素就地形变出可输入面板 + non-activating 焦点交接"在打包态/多 macOS 版本下不可靠到影响主流程 → 降级为 Approach A（独立捕获窗 + 侧展面板，焦点问题更简单）。
**核心假设的最快验证方式**：Stage 1 出可交互 HTML 原型演示"静止→hover 长成清单→热键形变出输入框→确认缩回"的完整形变与态信号；Stage 2 起手用最小 NSPanel spike 实测"全屏悬浮 + non-activating 写入 + 焦点回归"（见研究报告 Part 2 §5）。

### 演进路径

| 阶段 | 条件 | 方案形态 |
|------|------|---------|
| v1（当前） | 本地、单人、纯 GUI | 扁平无日期清单 + 双入口(热键写/hover 看) + 单形变胶囊 + 勾掉即清 |
| v2 | v1 验证"够用但想要一点结构" | 可选"今天/以后"两档 + 拖拽排序打磨 + 可选 `#标签` |
| v3 | 用户愿意让 AI 介入 | 可选"把一段话/一段会议记录拆成多条待办"（届时重启 AI Native 维度） |

**v1决策不堵死v2/v3的关键设计点**：v1 数据模型为"扁平条目流"，加一个可空的 `bucket`/`tags`/`due` 字段即可演进到 v2，不需重构；v1 不引入任何"必须依赖日期/项目"的交互，故 v2 增量是纯叠加。

## 被否定的方向

### 否定方向 D1：给待办加时间维度（日期 / 今天-以后 / NLP 解析）

- **考虑过的原因**：研究 MVP 含极小 NLP 日期解析（today/tomorrow/+3d）+ 实时 token 高亮，是顶级产品共识。
- **否定的原因**：用户 Phase 3 Q3 明确选"纯扁平清单，无日期"，并据此连带砍掉 `#标签`/`p1-p3`。日期/标签是"结构"，与"最简最轻"冲突。
- **来源**：用户 Q3 回答 + 研究 Part 1 §2 复杂度陷阱清单。

### 否定方向 D2：临时召唤式清单（Approach C）

- **考虑过的原因**：最克制、看长清单更舒展。
- **否定的原因**：与用户"hover 上去就地看待办"的直觉相左（Q2）。
- **来源**：用户初始诉求 +Q2 回答。

### 否定方向 D3：AI 自动分类/总结/排程待办（AI Native 层面否定）

- **考虑过的 AI 介入方式**：AI 自动给待办打标签、按优先级排序、生成今日计划。
- **否定的原因**：本产品 AI Native 判定为 not_suitable——这类介入引入"用户需判断 AI 对不对"的认知负担，与"瞄一眼即懂、绝不臃肿"正面冲突；且无证据支撑用户对此的接受度。
- **来源**：Phase 2.5 AI Native 评估（not_suitable）+ 研究 avoid-list（游戏化/智能化膨胀）。

## 假设前提

（均为用户 Phase 3 明确确认，非推断）

1. 工作中"快速写待办"的主路径是**热键召唤捕获**，hover 胶囊为辅看（Q2）。
2. 一条待办**无时间维度**——纯文字+勾选框+手动排序；无日期/标签/优先级/项目层级（Q3）。
3. 静止形态是**深色 Granola 式药丸**，顶部图标 + 下方未完成数（Q4 + 附图）。
4. 交互编排采用**单元素连续形变（灵动岛式，Approach B）**（Phase 4 选择）。
5. 勾掉一条 → 愉悦动效 → **几秒后自动清走**（纯 jotter，不堆积）（Phase 5 确认）。

## 我注意到你的设计思考方式

- 「他要很轻量很便捷。像 granola 一样」——你的锚点始终是**形态与体感（常驻、轻、不打扰）**，而非功能清单；这指引我把"简单"定义成一套坚决的非目标，而不是少做几个功能。
- 「不要把功能做复杂。要简单好用，轻量，动效好用，美观」——你把"动效"和"美观"与"不复杂"并列为同级目标；这让我把 reduced-motion 降级、形变态信号当成一等设计任务，而非合规脚注。
- （选 Q3）"纯扁平清单，无日期"——你在面对"要不要日期"时选了最激进的简化，确认了你要的是 **jotter 不是任务管理系统**。
